Claude Codeで非エンジニアが「できたこと・できなかったこと」正直レビュー

執筆： hybs02

Claude Codeレビュー非エンジニア実体験

Claude Codeのレビュー記事は「すごい」「革命」みたいな言葉で埋まりがちです。実際に手を動かした感想としては、確かに便利。でも万能ではない。ここでは、できたことと、できなかった（もしくは過信すると危ういと感じた）ことを、フラットに書きます。

できたこと

「日本語で頼む」がほぼ成立する

一番驚いたのはここです。「テキストファイルを置いたら記事になる仕組みを作って」とか「日本語が文字化けするから直して」みたいな、ふわっとした依頼でも、ちゃんと意図を汲んで動くものを返してくる。コマンドの暗記や文法の勉強は、ほぼ要りませんでした。

面倒な定型作業が一瞬で終わる

ファイルの一括処理、形式の変換、設定ファイルの修正。こういう「やれば終わるけど面倒」な作業は、明確に速い。半日で、ブログの生成からGitHubへの公開までひと通り通せました。

エラーで止まらない

途中で出たエラーも、原因を調べて直すところまで自分で進めてくれる。ここは、初心者が一番心折れるポイントなので、地味に効きました。

期待ほどではなかったこと

「丸投げで完璧」ではない

何を作りたいかが自分の中で曖昧なまま投げると、出てくるものもぼんやりします。良い結果は、良い指示から。結局、何を作りたいかを言語化する力はこちらに要求されます。ここを「AIが全部考えてくれる」と思い込むと、ズレた成果物に時間を取られます。

中身の良し悪しは人間が判断するしかない

文章でもコードでも、それっぽいものはすぐ出ます。でも「それっぽい」と「本当に良い」は別物。最終的な品質チェックは人間がやる前提で組まないと、薄いものを量産してしまう。実際このサイトでも、「公開前に人が必ず見直す」という関門をあえて入れています。

過信すると検証を飛ばす

スラスラ動くので、つい出力をそのまま信じたくなる。でも事実関係や細部は、自分で確かめる癖を残しておいたほうがいい。便利さと引き換えに、確認をサボる誘惑がある——これは道具のせいではなく、使う側の姿勢の問題です。

結論：相棒であって、代理人ではない

総じて、作業の9割を肩代わりしてくれる優秀な相棒。ただし、何を作るかを決め、出てきたものの良し悪しを判断する役割は、最後まで自分に残ります。そこを引き受ける気があるなら、これほど心強い道具もそうないと思います。

導入のしかたは Claude Codeの始め方 に、実際の構築記録は 自動更新ブログを作った全記録 にまとめています。

このブログについて

当サイトは「AIで自動更新するブログ」を実際に運営しながら、その記録を公開しています。仕組みの全工程はこちらの記事にまとめました。
